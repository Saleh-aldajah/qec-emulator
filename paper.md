\# Summary



`QEC Emulator` is an open-source Python package for constructing quantum

low-density parity-check (qLDPC) codes and benchmarking their decoders under

explicit, reproducible conditions. It implements the bivariate-bicycle (BB)

code family \[@Bravyi2024] together with small reference codes, and exposes

three decoders: belief propagation with ordered-statistics post-processing

(BP+OSD) through the `ldpc` package \[@Roffe2020; @ldpcPackage], minimum-weight

perfect matching (MWPM) through PyMatching \[@Higgott2023], and an exhaustive

lookup decoder. Noise can be sampled from phenomenological, repeated-cycle,

circuit-level, and hardware-model generators.



The package's defining feature is provenance. Every benchmark record stores the

SHA-256 digests of the parity-check matrices it used and a deterministic random

seed derived from the SHA-256 hash of the experiment context (code, decoder,

protocol, and physical error rate), so that any reported number can be

regenerated byte-for-byte and independently verified. The software provides a

Python API, a command-line interface, a FastAPI REST server for remote and

distributed execution, Docker images, multi-format export (CSV/JSON/HDF5), and

interoperability exports to Stim \[@Gidney2021], NetworkX, and `alist`/sparse

check-matrix formats. The source is released under the MIT license, with

continuous integration and an automated test suite.



\# Statement of need



Reproducible finite-instance benchmarking is a recognised weak point in the

qLDPC decoding literature. Progress on high-rate codes such as the

bivariate-bicycle family \[@Bravyi2024] has been rapid, but published decoder

comparisons are often difficult to reproduce: the precise code instance and

parity-check matrices are not always pinned, the rule used to reduce a qLDPC

syndrome to a graph for matching decoders is frequently left implicit,

random-seed policies are rarely documented, and data provenance is seldom

machine-verifiable. Mature single-purpose libraries address parts of the

pipeline — `ldpc` provides BP+OSD \[@Roffe2020], PyMatching provides fast

matching \[@Higgott2023], and Stim provides circuit-level stabilizer simulation

\[@Gidney2021] — but none ties code construction, decoder representation, noise

model, seed policy, and exact matrix bytes to every emitted result in a single,

inspectable workflow.



`QEC Emulator` is designed around that gap. It standardises deterministic

seeding (SHA-256-derived, avoiding Python's process-dependent hash

randomisation), attaches SHA-256 matrix digests and a file-level manifest to

every run, ships verification scripts that re-check the manifest and a

finite-instance code certificate, and pins its software environment. Because a

graphlike decoder for qLDPC codes requires reducing weight-three syndrome

columns to edges — a step that can discard correlations — the package exposes

that reduction explicitly through its interoperability layer, so the same code

instance can be moved between the BP+OSD and matching ecosystems with a

documented, reviewable graphification rather than an undocumented one. The

intended users are quantum-error-correction researchers who benchmark qLDPC

decoders, particularly on BB codes; reviewers who need to reproduce reported

numbers without reconstructing an undocumented pipeline; and instructors

teaching fault-tolerance concepts who benefit from a single tool spanning code

construction, decoding, noise, and provenance.



\# Functionality



The package is organised in five layers: code construction (`codes.py`: BB and

reference codes with SHA-256 pins), decoders (`decoders.py`: BP+OSD, MWPM,

lookup), noise models (`noise.py`: phenomenological, repeated-cycle,

circuit-level, hardware), benchmark drivers (`benchmark.py`: parameter sweeps,

repeated-cycle runs, threshold scans), and provenance (`provenance.py`).

Version 3.0 adds YAML/JSON configuration, parallel job scheduling, multi-format

export, an interoperability module (Stim/DEM, NetworkX Tanner graphs,

`alist`/sparse), a visualisation module, a FastAPI REST server, and a `Typer`

command-line interface. The project ships a `Dockerfile` and `docker-compose`

configuration, continuous integration, and an automated test suite. It depends

on `ldpc`, `pymatching`, and `numpy`, with optional `stim` and `h5py`.



A minimal benchmark sweep, fully reproducible from its recorded seeds and

matrix digests, can be launched from the command line

(`qec-emulator sweep --code BB72 --decoder bposd --trials 5000`), from the

Python API, or as an asynchronous job through the REST server. The BB72

certificate is verified with `qec-emulator verify`.



\# Research enabled



The suite underpins a companion study that benchmarks BP+OSD against two

explicitly graphified reference decoders on the BB72, BB144, and BB288 codes,

verifies a finite-instance certificate for BB72, and organises the resulting

evidence with an adversarial-channel-equilibria framework. All primary

datasets, runner scripts, and the SHA-256 manifest are archived

\[@AlDaajeh2026zenodo].
